## Требования

- [Git](https://git-scm.com/)
- [Docker](https://www.docker.com/)
- [Docker Compose](https://docs.docker.com/compose/)

## Инструкция запуска проекта

### 1. Клонируйте данный репозиторий и перейдите в каталог проекта

```bash
git clone https://github.com/RomanSrgvchL/DocumentService.git
cd DocumentService
```

### 2. Настройте переменные окружения

```bash
cp .env.example .env
```

> Файла `.env.example` достаточно для локального запуска. При необходимости измените значения переменных в файле `.env`.

### 3. Запустите контейнеры через Docker Compose

```bash
docker compose up -d
```

> Dockerfile использует multi-stage, сборка сервиса выполнится автоматически.

### 4. Следите за логами

> Откройте контейнер document-service в Docker Compose Desktop, все логи будут отображаться в консоли.

> *Для локального запуска в IDE предварительно остановите контейнер document-service и укажите ваш .env файл в конфигурации ранера. Процесс выполнения также можно будет посмотреть по логам из консоли.

### 5. Дополнительная информация

> Сортировка поиска документов осуществляется по одному из полей: ID | USERNAME | CREATED_AT | UPDATED_AT
>
> Также можно передать направление сортировки (ASC | DESC)

### 6. Ответы на вопросы

> Что бы вы поменяли, чтобы обработка одного запроса уверенно работала с
5000+ id?
> > Я бы использовал пакетное получение/обновление для повышения эффективности, так как при большом количестве документов много времени бы уходило на сетевые задержки между запросами к БД.

> Как бы вы вынесли реестр утверждений в отдельную систему (отдельная БД или
отдельный HTTP-сервис)?
> > Я бы вынес реестр утверждений в отдельный микросервис с собственной БД, а между микросервисами выстроил бы асинхронное взаимодействие (например, по Kafka). Сообщения о регистрации отправлял бы через по outbox паттерну для обеспечения гарантий доставки.
